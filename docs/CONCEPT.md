## MissionOrch — 项目概念与设计文档

### 一、项目定位

MissionOrch 是一个**多智能体任务编排系统**，核心目标是：

> 接收一段自然语言的**作战任务描述**，通过多智能体协作循环，自动生成**战役级行动方案（COA）**，最终以结构化格式交付给下游系统（想定生成 Agent）。

它解决的核心问题是：**将人类指挥员的作战意图，转化为可被仿真系统消费的结构化行动方案。**

---

### 二、核心概念

#### 2.1 COA（Course of Action，行动方案）

COA 是军事决策过程（MDMP）中的核心产物。本系统生成的 COA 是**战役级**的，聚焦四要素：

| 要素 | 含义 | 在矩阵中的体现 |
|------|------|----------------|
| **Who**（谁做） | 执行主体、指挥链 | 矩阵纵轴：作战单元 |
| **What**（做什么） | 任务内容、预期效果 | 矩阵单元格：行动列表 |
| **When**（何时做） | 阶段划分、转换条件 | 矩阵横轴：阶段 + 转换触发条件 |
| **Where**（何地做） | 作战区域、地理位置 | 行动描述中隐含 |

**不涉及的内容**（留给下游想定生成系统）：
- 具体飞行高度、速度、航线（How）
- 武器参数、弹药消耗量
- 精确到秒的时间表
- 平台数量、编队构型

#### 2.2 COA 矩阵

COA 的核心表达形式是**"作战单元 × 阶段"二维矩阵**：

```
              阶段1              阶段2              阶段3
            (转换条件1)        (转换条件2)        (转换条件3)
单元A     [行动1, 行动2]     [行动1, 行动2]     [行动1, 行动2]
单元B     [行动1, 行动2]     [行动1, 行动2]     [行动1, 行动2]
单元C     [行动1, 行动2]     [行动1, 行动2]     [行动1, 行动2]
```

- **纵轴（作战单元）**：参与行动的各类兵力/平台，附带职能角色描述
- **横轴（阶段）**：作战行动的时序划分，每个阶段有名称、目标和**转换触发条件**
- **单元格（行动）**：该单元在该阶段的 2-4 个具体行动描述

#### 2.3 阶段转换：条件驱动而非时间驱动

战役级 COA 的阶段转换**不使用固定时间窗口**，而是使用**条件/事件触发**：

| 方式 | 示例 | 适用场景 |
|------|------|----------|
| ❌ 固定时间 | "H+05min ~ H+20min" | 战术级（分钟级别的精确行动） |
| ✅ 条件触发 | "敌防空网络被压制后" | 战役级（依赖态势发展） |
| ✅ 事件驱动 | "制空权确立后" | 战役级（依赖前序阶段成果） |
| ✅ 混合 | "D+1 或 桥头堡建立后（以先到者为准）" | 战役级（时间兜底 + 条件优先） |

**设计理由**：
- 战役级行动的持续时间高度不确定，受敌情、天候、后勤等多因素影响
- 阶段转换的本质是"前序条件达成"，而非"时钟到点"
- 条件驱动更符合军事决策的实际逻辑（"当 X 达成时，转入 Y 阶段"）
- 下游想定生成系统可以根据条件设计仿真中的事件触发器

#### 2.4 辅助要素

除矩阵外，COA 还包含：

- **效果链（Effects Chain）**：期望达成的战略效果及其衡量指标
- **决策点（Decision Points）**：需要指挥员介入决策的关键节点（if-then 逻辑）
- **关键风险（Critical Risks）**：已识别的风险及缓解措施

### 三、系统架构

#### 3.1 多智能体循环

```
任务描述(自然语言)
       │
       ▼
  ┌─────────┐
  │ Planner │──→ 自然语言 COA 矩阵表格
  └─────────┘
       │
       ▼
  ┌─────────┐
  │  Judge  │──→ 评分 + 反馈（JSON）
  └─────────┘
       │
       ▼
  ┌───────────┐
  │ Reflector │──→ 改进建议（自然语言）
  └───────────┘
       │
       ▼
  ┌─────────┐
  │ Planner │──→ 改进后的 COA 矩阵表格
  └─────────┘
       │  (循环直到评分达标或达到最大迭代次数)
       ▼
  ┌────────────────┐
  │ COATableParser │──→ 结构化 COA 对象（Pydantic）
  └────────────────┘
       │
       ▼
  ┌────────────────┐
  │ COATransformer │──→ JSON / YAML / 之前得到的自然语言扁平矩阵
  └────────────────┘
```

**关键设计决策**：整个循环中 Agent 之间传递的始终是**自然语言 Markdown 表格**，不是 JSON。只有在最终输出阶段，才由 `COATableParser` 将表格解析为结构化的 `COA` 对象。

**理由**：
- LLM 生成和理解自然语言表格比生成严格 JSON 更稳定
- 自然语言表格对 Judge 和 Reflector 来说更容易评估和反思
- 避免了循环中反复 JSON 序列化/反序列化的格式错误风险

#### 3.2 核心模块

| 模块 | 文件 | 职责 |
|------|------|------|
| **Schema** | `schemas/coa.py` | Pydantic V2 数据模型：Phase、Unit、Action、Effect、DecisionPoint、COA |
| **Planner** | `core/agent_impl.py` + `prompts/planner.txt` | 生成自然语言 COA 矩阵表格 |
| **Judge** | `core/agent_impl.py` + `prompts/judge.txt` | 评估 COA 质量，输出评分和反馈 |
| **Reflector** | `core/agent_impl.py` + `prompts/reflector.txt` | 分析缺陷，提供改进建议 |
| **Parser** | `core/coa_parser.py` | 将自然语言表格解析为结构化 COA 对象 |
| **Transformer** | `core/coa_transformer.py` | 将 COA 对象转换为 JSON/YAML/扁平矩阵等格式 |
| **Orchestrator** | `core/orchestrator.py` | 编排多智能体循环 |
| **Model Router** | `utils/model_router.py` | 多模型提供商路由（OpenAI/Gemini/Claude等） |
| **RAG** | `utils/rag.py` | 知识检索增强 |

### 四、输入与输出格式

#### 4.1 输入：任务描述

系统接收一段**自然语言的作战任务描述**，例如：

```
对敌方沿海防御体系实施联合突击行动。
敌方在沿海部署了多层防空体系，包括远程预警雷达、中程地空导弹阵地和近程防空系统。
我方可用兵力包括：电子战飞机编队、制空战斗机中队、精确打击编队、预警指挥机、空中加油机。
作战目标：压制敌防空体系，夺取局部制空权，摧毁敌指挥通信节点。
```

输入不需要任何特定格式，系统会通过 RAG 检索相关知识来辅助规划。

#### 4.2 输出：多格式 COA

系统输出包含以下内容：

**主要输出 — 自然语言 COA 矩阵表格**（Markdown）：

```markdown
| 作战单元 / 职能 | 阶段1: 电磁压制 *(敌预警雷达被致盲后→转阶段2)* *(瘫痪敌态势感知)* | 阶段2: ... |
| :--- | :--- | :--- |
| 电子战飞机 *(电磁掩护)* | 1. 压制敌预警雷达 2. 投放诱饵 | 1. ... |
```

**结构化输出**（由 Parser 从表格转换而来）：

- **JSON 格式**：完整的 COA 对象序列化
- **YAML 格式**：人类可读的配置格式
- **扁平矩阵格式**：`{ "matrix": { "unit_id": { "phase_id": ["action1", ...] } } }`
- **压缩格式**：包含关键指标的概览

#### 4.3 输出示例（扁平矩阵格式）

```json
{
  "phases": [
    {
      "phase_id": "Phase_1",
      "name": "电磁压制与突防",
      "transition_trigger": "敌预警雷达网络被压制，安全走廊建立",
      "objective": "瘫痪敌态势感知能力"
    }
  ],
  "units": [
    {
      "unit_id": "EA_18G",
      "name": "电子战飞机编队",
      "role": "电磁掩护"
    }
  ],
  "matrix": {
    "EA_18G": {
      "Phase_1": [
        "实施伴随干扰，压制敌预警雷达",
        "投放诱饵弹模拟大机群进入"
      ]
    }
  }
}
```

### 五、COA → 想定转化思路

本系统生成的 COA 是**中间产物**，后续需要交付给想定生成系统（是另一个 Agent 系统，所以这里只是了解，不需要实现）转化为可执行的仿真想定。

#### 5.1 转化链路

```
MissionOrch 输出的 COA（结构化 JSON）
       │
       ▼
┌──────────────────────┐
│  想定生成 Agent 系统  │
│                      │
│  1. 解析 COA 矩阵    │
│  2. 补充战术细节      │  ← 这一步由想定 Agent 完成
│  3. 生成仿真实体      │
│  4. 设置事件触发器    │
│  5. 输出想定文件      │
└──────────────────────┘
       │
       ▼
  仿真系统可执行的想定文件
```

#### 5.2 COA 与想定的映射关系

| COA 要素 | 想定中的对应 | 转化说明 |
|----------|-------------|----------|
| **作战单元（Unit）** | 仿真实体/兵力编组 | 需要补充：具体平台数量、初始位置、编队构型 |
| **阶段（Phase）** | 事件触发器 + 阶段状态机 | 转换条件 → 仿真事件触发器（如"雷达被摧毁"事件） |
| **行动（Action）** | 实体行为序列 | 需要补充：具体航线、速度、高度、武器参数 |
| **效果链（Effect）** | 仿真评估指标 | 转化为仿真中的评估条件和统计指标 |
| **决策点（Decision Point）** | 分支逻辑 / 人在回路 | 转化为仿真中的条件分支或暂停点 |

#### 5.3 为什么 COA 不应包含战术细节

COA 故意**不包含**战术级细节（飞行高度、速度、武器参数等），原因是：

1. **职责分离**：COA 回答"做什么"，想定回答"怎么做"
2. **灵活性**：同一个 COA 可以根据不同的兵力配置生成不同的想定
3. **可复用性**：COA 是战役级的决策产物，不应绑定到具体的仿真平台
4. **Agent 自主性**：想定生成 Agent 应该有自主权来决定战术细节，而不是被 COA 硬编码

#### 5.4 条件驱动阶段对想定生成的意义

COA 中的**条件触发式阶段转换**对想定生成特别重要：

```
COA 中的条件：                    想定中的实现：
"敌防空网络被压制后"        →     事件触发器：当敌方雷达存活率 < 30% 时
"制空权确立后"              →     事件触发器：当空域内敌机数量 = 0 时
```

### 六、设计约束与边界

| 约束 | 说明 |
|------|------|
| **抽象层级** | 战役级（Operational），不涉及战术级（Tactical）细节 |
| **输出消费者** | 下游想定生成 Agent 系统，不是人类操作员直接使用 |
| **循环传递格式** | Agent 间传递自然语言 Markdown 表格，不传 JSON |
| **结构化转换** | 仅在最终输出时由 COATableParser 执行一次转换 |
| **阶段转换** | 条件/事件驱动，不使用固定时间窗口 |
| **模型无关** | 支持多种 LLM 后端（OpenAI/Gemini/Claude/Seed/Qwen/Deepseek等） |
